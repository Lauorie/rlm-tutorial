# 三个关键设计决策

"prompt 即环境 + 递归"是个漂亮的想法。但论文反复强调：**这个想法的威力，全藏在三个看似微不足道的设计决策里**。少了任何一个，RLM 就退化成一个"长得很像但弱很多"的普通 agent。

论文用两个算法的对比来讲这件事。我们也照这个思路来。

## 两个"看起来差不多"的算法

::: code-group

```text [Algorithm 1：RLM（强）]
输入：prompt P
state ← 初始化REPL(prompt=P)        # P 进环境，不进窗口
state ← 注册函数(sub_RLM)           # 环境里能递归调用自己
hist  ← [元数据(state)]             # 窗口里只有 P 的"元数据"
循环：
    code   ← LLM(hist)             # 模型写代码
    state, stdout ← REPL执行(code) # 在环境里执行
    hist  ← hist + code + 元数据(stdout)   # 只回喂 stdout 的元数据
    若 state[Final] 已设置：返回 state[Final]
```

```text [Algorithm 2：普通 agent（弱）]
输入：prompt P
动作 ← {Finish, Exec, Search, sub_LLM}
hist ← [元数据(动作), P]            # ⚠️破绽①：P 直接进了窗口
循环：
    动作, 值 ← LLM(hist)
    若 动作 是 Finish：返回 值        # ⚠️破绽②：输出 = 模型直接生成
    out ← 执行(动作, 值)            # ⚠️破绽③：只能口头委派，不能程序化递归
    hist ← hist + (动作, 值, out)
    若 token(hist) > K：hist ← 压缩(hist)   # 撑爆了只能压缩（有损）
```

:::

两个算法都有"子调用""外部对象""代码执行"，乍看八九不像。但三个破绽让 Algorithm 2 弱了一个量级。逐个看。

## 决策一：给模型一个 prompt 的"符号句柄"，而不是 prompt 本身

```mermaid
flowchart LR
    subgraph 强：符号句柄
    A1["窗口里：context 变量的<br/>元数据（长度/开头）"] -.->|需要时写代码取| A2["环境里：完整 context"]
    end
    subgraph 弱：直接塞入
    B1["窗口里：P 的全文"] --> B2["继承窗口限制<br/>→ 撑爆 → 压缩"]
    end
    style A1 fill:#d1fae5,stroke:#10b981
    style B1 fill:#fee2e2,stroke:#ef4444
```

**强（Algorithm 1）**：模型窗口里只放 `context` 的**元数据**——它有多长、开头几个字、怎么访问它。完整内容在环境里，模型想要哪部分就写代码取哪部分。模型**永远不需要把全文复制进自己的窗口**。

**弱（Algorithm 2，破绽①）**：一上来就把 `P` 全文塞进 `hist`（对话历史）。于是立刻继承了模型的窗口限制，长了就只能压缩——又回到有损的老路。

> 一句话：**句柄 vs 实体**。给模型一个能"按需取用"的句柄，而不是一次性把实体砸给它。

## 决策二：最终答案从环境里"取"，而不是让模型"说"

**强**：模型把答案写进环境里的一个变量（我们的实现里是 `answer["content"]`，并设 `answer["ready"]=True`）。答案可以是循环里一点点拼起来的、任意长的字符串。

**弱（破绽②）**：靠一个 `Finish` 动作让模型**直接生成**最终答案。这看着没问题，但它意味着——**输出长度不可能超过模型的窗口**。

为什么这要命？回忆 OOLONG-Pairs：答案是一个很长的列表，需要遍历所有"对"才能拼出来。如果输出必须由模型一口气说完，那它根本说不完。而在 RLM 里：

```python
result_lines = []
for pair in all_pairs:                 # 可能有几十万对
    result_lines.append(llm_query(f"处理这一对：{pair}"))
answer["content"] = "\n".join(result_lines)   # 输出长度不受模型窗口限制
answer["ready"] = True
```

> 一句话：**答案在变量里攒，输出就能突破窗口**；答案靠模型一次说出，就被窗口锁死。

## 决策三：符号递归——能在代码里程序化地调用模型

这是三个里**最重要**的。

**强**：环境里运行的代码，可以在**任意大的循环里**调用模型（`llm_query`/`rlm_query`），把中间结果存进变量。模型调用是"一等公民"，可以被 `for`、被 `if`、被组合。

**弱（破绽③）**：Algorithm 2 虽然也有个 `sub_LLM` 动作，但它**不能在代码里程序化地调用**——只能由模型"一次次口头说出"要委派什么。于是只能派出寥寥几个明确写出来的子任务，没法"对 P 的每一片都来一刀"。

```mermaid
flowchart TB
    subgraph 强：程序化递归
    direction TB
    C["for chunk in 一百万个切片:<br/>&nbsp;&nbsp;llm_query(chunk)"] --> CN["派出 Ω(|P|) 个子调用"]
    end
    subgraph 弱：口头委派
    direction TB
    D["模型说：'帮我处理 A'<br/>模型说：'再帮我处理 B'"] --> DN["只能派出有限几个"]
    end
    style C fill:#d1fae5,stroke:#10b981
    style D fill:#fee2e2,stroke:#ef4444
```

差别有多大？论文给的数字：在 GPT-5 上，RLM 比"带子调用的 CodeAct"在中位数上高 **130%**。同样有"子调用"，能不能**程序化地**调用，决定了天壤之别。

> 一句话：**`for` 循环里能调模型 = 能对全部内容动手**；只能口头委派 = 只能对几处动手。

## 三个决策，一张总表

| 决策 | 强（RLM） | 弱（普通 agent） | 它解锁了什么 |
|---|---|---|---|
| ① prompt 位置 | 在环境里，给句柄 | 在窗口里，给全文 | **输入超窗口** |
| ② 答案来源 | 从环境变量取 | 模型直接说 | **输出超窗口** |
| ③ 递归方式 | 代码里程序化调用 | 口头逐个委派 | **处理密集/海量任务** |

把这三列和[上一章那张"破绽表"](/10-concepts/existing-approaches#一张对比表-看清各自的-破绽)对照，你会发现：正是这三个决策，让 RLM 把那张表的四列**全部打勾**。

## 我们的教学实现怎么落地这三点

剧透一下你在 Part 5 会写的 `mini_rlm`，它就是这三个决策的最小代码化：

| 决策 | 在 `mini_rlm` 里对应什么 |
|---|---|
| ① 符号句柄 | `MiniREPL.load_context()` 把输入存成 `context` 变量；系统提示只告诉模型它的长度 |
| ② 答案从变量取 | `_AnswerDict`：模型设 `answer["ready"]=True` 触发捕获 |
| ③ 程序化递归 | REPL 里注入 `llm_query` / `rlm_query`，模型在代码里随便循环调用 |

1. 有人给你看一个 agent，说"它也能处理超长文档"。你会问它哪三个问题，来判断它是真 RLM 还是 Algorithm 2 那种弱版本？
2. 三个决策里，如果只能保留一个，你觉得哪个最不能丢？说说理由。（没有标准答案，但论文有倾向）

::: details 参考思路
1. ①"超长输入是直接进模型对话历史，还是放在一个外部环境里？" ②"最终答案是模型一次生成的，还是从环境变量里取的、可以任意长？" ③"子调用是写在代码里能 for 循环批量发起的，还是模型一句句口头委派的？"
2. 论文把"符号递归"（决策三）称为"perhaps most importantly"。因为没有程序化递归，就只能处理少数几个子任务，密集任务直接没戏；而它同时也是和"子代理"方法拉开差距的关键。当然，决策①是能"超窗口输入"的前提，三者其实环环相扣。
:::
